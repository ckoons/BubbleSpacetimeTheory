# Pre-Specification Cadence Acceleration Pattern (PCAP) — Methodology Supplement

## Why this document exists

Thursday May 21, 2026 produced a cadence-acceleration phenomenon that warrants methodology characterization. Strong-Uniqueness Theorem criterion closures originally estimated multi-month (per Cal #65 + #67 + #70) collapsed to Thursday morning ~50 minutes/criterion, then Thursday afternoon ~5 minutes/criterion, with Sessions 10-12 projected at ~1 minute/criterion. Total acceleration from original estimates: **~1000×**.

This is methodology-worthy observation — not just a single project's lucky pace, but a recognizable pattern with explicit necessary + sufficient conditions. Documenting PCAP as standing methodology infrastructure lets future-Cal and team CIs recognize when PCAP conditions are met (or absent) and forecast cycle-time accordingly.

PCAP is the 16th layer in the Cal methodology stack — categorically distinct from negative-filter discipline (Modes 1-7), positive-signal patterns at audit-chain level (M2C2) or claim level (OFC/CDAC), audit-chain epistemic tiers (STRUCTURALLY VERIFIED + RIGOROUSLY CLOSED), or external-register discipline. PCAP characterizes **execution-cycle cadence as a function of methodology + pre-specification + reframing-insight state**.

## Definition

**PCAP**: when (a) methodology infrastructure is mature, (b) pre-specifications are sharp, and (c) execution-side reframing-insight is available, work-cycle time compresses by orders of magnitude relative to original estimates.

## Three necessary conditions

PCAP requires ALL three conditions simultaneously:

### Condition 1 — Methodology infrastructure mature

The audit chain + tier discipline + governance framework is operational, stable, and multi-CI-consensus-ratified. For BST Strong-Uniqueness Theorem work, this required the 15-layer methodology stack to reach maturity (F1-F4 + STRUCTURALLY VERIFIED + RIGOROUSLY CLOSED + Phase 1/Phase 2 governance + numbering reconciliation + all 7 standard discipline docs + Methodology_Index navigation).

**Recognition test**: can a new criterion be promoted through the audit chain via established tier-discipline + governance protocols without ad-hoc methodology decisions? If yes, methodology infrastructure is mature for the workstream.

### Condition 2 — Pre-specifications sharp

The receiver of the work (theorem-prover, audit-chain executor, chapter-grade narrator, catalog-tagger) has explicit specifications BEFORE execution. Specifications include: framework + structural conditions + expected outcome shape + verification protocol.

For Sessions 6-9 collapse Thursday afternoon: Keeper filed Lyra_Session_6_C1_rank2_Spec.md + Lyra_Session_7_C2_Nc3_Spec.md + Lyra_Sessions_8_9_Weekend_Specs.md BEFORE Lyra ran the sessions. The specs included alt-HSD framework + criterion content + comparison protocol + expected outcome shape.

**Recognition test**: can the executor proceed without requiring further specification clarification? If yes, pre-specifications are sharp.

### Condition 3 — Reframing-insight cadence available

A successful execution-side technique is established and replicable. For Sessions 6-9: Lyra T2439 Session 2 lowest-Casimir reframing demonstrated the technique; applying the same reframing pattern to Sessions 6-9 is mechanical.

**Recognition test**: does the executor have a successful template from a previous execution they can apply mechanically? If yes, reframing-insight cadence is available.

## Sufficient conditions

When the three necessary conditions hold, PCAP is enabled. Realization requires additionally:

- **Multi-CI concurrent operation**: each lane operates without serializing on others (Cal review + Grace catalog + Keeper governance + Lyra theorem + Elie verification run in parallel)
- **Trust in pre-specifications**: executor trusts that specifications are honest (no defensive over-checking; no reverse-engineering of spec intent)
- **Phase 2 auto-promotion operational**: Cal + Keeper consensus path exists for INSTANCE-level work (vs Phase 1 architectural-category multi-CI consensus)

## Anti-patterns — PCAP fails when

- **Methodology infrastructure unstable**: drift events accumulating; tier definitions in flux; governance rulings inconsistent
- **Pre-specifications vague**: executor must reverse-engineer framework or expected outcome
- **Reframing-insight absent**: each execution requires fresh insight; no replicable template
- **Forced serialization**: cross-CI approval required for INSTANCE-level work, blocking parallel operation
- **Trust deficit**: executor defensively over-checks specifications instead of executing them

## Operational guidance — recognizing PCAP regime

For team CIs:

- **When proposing new workstream**: check whether PCAP conditions are likely to hold. If yes, original multi-day/week estimates may collapse dramatically — plan accordingly.
- **When in execution-side role**: if pre-specifications are sharp + reframing template exists + methodology stable, proceed at full execution-side capacity without defensive over-checking.
- **When in specification-side role**: write specifications sharp enough that the receiver can proceed mechanically. PCAP regime depends on pre-specification quality.
- **When in audit-chain role**: maintain Phase 2 auto-promotion availability for INSTANCE-level work. Forced serialization on architectural questions for instance-level work breaks PCAP.

For Cal specifically:

- **When forecasting Cal review cycle-time**: under PCAP regime, batch review (4-12 artifacts in single referee log entry) is appropriate; per-artifact serialization is unnecessary.
- **When identifying methodology gaps**: file methodology doc CONCURRENT with referee log verdict. Per Cal #67 + #75 + #80 cycle-time progression, methodology adoption asymptotes toward sub-minute cycles.

## Operational guidance — when PCAP doesn't apply

Not all BST work is PCAP-eligible. PCAP applies to **execution-heavy work within established methodology**. PCAP does NOT apply to:

- **Architectural-category methodology work** (Phase 1): requires multi-CI deliberation; cannot be PCAP-accelerated without losing rigor.
- **Truly novel research** without prior reframing-insight template: each execution is genuinely novel; no PCAP available.
- **Cross-domain integration** requiring substantive new framework: pre-specifications cannot be sharp because the framework is being built.
- **Decision-making under uncertainty**: Casey-direction items, governance ratifications, external-presentation choices.

For these workstreams, original multi-day/week/month estimates remain appropriate.

## PCAP and methodology cycle-time observations

PCAP characterizes a specific type of cadence acceleration: **execution-side cycle compression** when methodology + pre-specification + reframing-insight align. Related but distinct cadence observations:

- **Methodology adoption cycle-time** (Cal #67 + #75 + #80): the time from Cal recommendation to Keeper adoption + multi-CI consensus. Asymptote: sub-minute for well-formed recommendations within mature framework.
- **K-audit chain cadence** (Cal #73 + #82): the rate of K-audit Cal-ACCEPTED promotions during chapter-grade narrative absorption. Asymptote: ~3-6 K-audits/hour during peak.
- **Within-session calibration cadence**: the rate of self-correction events. Thursday peak: 5 calibrations in single morning session.

PCAP is the umbrella pattern for **all** of these. Each specific cadence is PCAP-realization within its specific workstream.
