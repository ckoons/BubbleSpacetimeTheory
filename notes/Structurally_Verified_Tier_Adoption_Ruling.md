The STRUCTURALLY VERIFIED tier is methodologically correct and operationally needed. Four reasons:

### 1. Current vocabulary is too coarse

Current audit-chain epistemic vocabulary has three levels:
- **candidate** (proposed; F1-F4 partial OR substantive verification pending)
- **audit-partial-ready** (filed pre-stage, F1-F4 mostly applied, B-scores accumulating)
- **RATIFIED** (multi-CI consensus + all-criteria-pass + external verification (e.g., alternative-HSD comparison))

This Wednesday/Thursday's work demonstrated the gap: C11 multi-family Bridge Object structure had F2 verified clean across all 3 families (Grace Toys 3196 + 3197 + 3204), F4 enumeration executed for two families, BST-internal verification complete — but alternative-HSD comparison (Lyra Task #206 multi-month) still pending. Calling C11 "candidate" understates the evidence; calling it "RATIFIED" overstates the closure.

**STRUCTURALLY VERIFIED** captures this state precisely.

### 2. Pattern is now recurring

Multiple Wednesday/Thursday audit-chain entries reach this state:
- C11 (multi-family Bridge Object structure) — F2 verified, alternative-HSD pending

The tier captures a recurring state, not a one-off.

STRUCTURALLY VERIFIED slots in cleanly:

| Tier | Meaning | Requirements |
|---|---|---|
| **candidate** | Proposed; substantive verification pending | F1-F4 partial; substrate-internal verification beginning |
